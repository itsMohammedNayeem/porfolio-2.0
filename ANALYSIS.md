# Portfolio 2.0 — Repo Audit & Modernization Plan

_Audit date: 2026-07-07 · Audited at commit `cffcdff` (main) · Node v22.18.0 / npm 10.9.3_

## 1. Current Architecture

**Stack:** Next.js 14.0.4 (App Router + leftover Pages Router API routes), React 18, TypeScript 5, Tailwind CSS 3.4, framer-motion 10, Sanity v3 (embedded Studio + standalone studio project), deployed on Vercel.

### Component tree

```
app/layout.tsx            — RootLayout (Inter font, metadata)
app/page.tsx              — Home (async server component; fetches all content)
├── Header                — social icons (react-social-icons), framer-motion entrance
├── Hero                  — typewriter (react-simple-typewriter), profile img, nav buttons
│   └── BackgroundCircles — animated radial rings
├── About                 — profile pic + background text
├── WorkExperience        — horizontal snap-scroll list
│   └── ExperienceCard    — per-job card (logo, tech icons, points)
├── Skills
│   └── Skill             — icon + hover proficiency %
├── Projects              — horizontal snap-scroll project cards
└── ContactMe             — react-hook-form → mailto: link
app/studio/[[...index]]/page.tsx — embedded Sanity Studio at /studio
```

### Data flow (current)

```
app/page.tsx (server component)
  → fetch(`${NEXT_PUBLIC_BASE_URL}/api/get*`)     ← HTTP round-trip to itself
    → pages/api/get*.ts (5 API routes)
      → sanityClient.fetch(groq`*[_type == "…"]`)  ← Sanity project 79vh6zyc / production
```

This self-fetch pattern (server component → own API route → Sanity) is a known
anti-pattern from the tutorial this site was based on. It requires
`NEXT_PUBLIC_BASE_URL` to be set correctly per environment, adds five serverless
functions, and doubles latency. The server component could (and should) query
Sanity directly — or the client could, since the dataset is public.

### Sanity CMS

- **Project:** `79vh6zyc`, dataset `production` (public read; project IDs are not secrets).
- **Schemas** (`sanity/schemas/`): `pageInfo`, `experience`, `skill`, `project`, `social`.
- **Two studios exist:** an embedded one at `/studio` (bundled into the Next app) and a
  standalone studio project in `sanity/` with its own `package.json` (older deps:
  styled-components 5, TS 4, prettier 2). They duplicate each other.
- `sanity/.env` is committed — it contains only the public project ID/dataset, **no secrets**.

### Styling

- Tailwind 3.4 with a custom palette (`sun` #D92815, gray/slate scale), custom
  `CascadiaMono` font (woff2 in `public/fonts`), `tailwind-scrollbar` plugin,
  `.heroButton` / `.contactInput` component classes in `globals.css`.
- **styled-components is NOT used anywhere in app code.** It is a direct dependency
  only because Sanity Studio v3 peer-depends on it. There is no real "dual styling
  setup" to consolidate — removing the embedded Studio removes the dependency.

### Other observations

- Images use plain `<img>` with Sanity CDN URLs — `next/image` is **not** used, so
  no Next-specific image optimization is being lost by migrating.
- SEO metadata is minimal and partly boilerplate: description is literally
  `"Generated by create next app"`. No Open Graph/Twitter tags, no favicon config.
- Dead/vestigial files: `test.json` (empty), commented-out `getStaticProps` in
  `app/page.tsx`, commented-out hardcoded social icons in `Header.tsx`, duplicate
  ESLint configs (`.eslintrc` legacy airbnb config + `.eslintrc.json` next config —
  neither works with the installed ESLint 9).

Items 2, 5 and 6 are almost certainly why **Vercel deploys fail**: the build breaks on
the TS error, and even if it compiled, build-time page rendering would crash without
`NEXT_PUBLIC_BASE_URL` and `NEXT_PUBLIC_SANITY_PROJECT_ID` env vars set in Vercel.

### 🟠 Security — `npm audit`: 37 vulnerabilities (2 critical, 11 high, 22 moderate, 2 low)

- **Critical:** `next` 14.0.4 (SSRF in Server Actions, cache poisoning, image-optimization DoS — fixed only in much newer Next); `form-data` (CRLF injection).
- **High:** `lodash`, `ws`, `tar-fs`, `braces`, `cross-spawn`, `minimatch`, `picomatch`, `flatted`, `@typescript-eslint/*`, `vite` (transitive via Sanity tooling).
- Most highs are in dev/tooling dependency chains (Sanity CLI, ESLint), but `next` itself is runtime-critical. **A clean audit is impossible without either upgrading Next to current or leaving Next** — this ties Goal 2 to the framework decision.

### 🟡 Quality / a11y / responsiveness (to fix in later PRs)

- `metadata.description` is create-next-app boilerplate; no OG/Twitter cards, no favicon.
- `ExperienceCard` company logo `<motion.img>` has no `alt`; `Projects` image has empty `alt`; `Skill` img has hardcoded `alt="Tailwind CSS"` for every skill.
- Buttons inside `<Link>` (`Hero`, `page.tsx` footer) — invalid interactive nesting for keyboard/AT users.
- `text-[min(3vw, 100px)]` etc. contain a space, so Tailwind never generates these classes (silent no-ops); `overflow-y-scrollbar-track-gray-400/20` in `About.tsx` is not a real class.
- Contact form does no validation despite react-hook-form being present; submit builds an unencoded `mailto:` URL (`encodeURIComponent` missing → subjects/messages with `&` break).
- Fixed-height hero text (`h-[20px]` on a multi-size `h1`) and heavy `tracking-[15px]/[20px]` headings overflow on small phones.
- Empty-string `url=''` on the email SocialIcon in `Header.tsx`.
- Dead code listed in §1; `.worklog`-style repo hygiene: `test.json` is empty.

## 3. Framework Decision

### What Next.js is actually providing today

| Next feature | Used? | Notes |
|---|---|---|
| SSR/SSG | ⚠️ Nominal | One server component whose only job is to call its own API routes; no ISR/revalidation active |
| API routes | ✅ (5) | Pure pass-throughs to a **public** Sanity dataset — add latency and env-var fragility, no security benefit |
| `next/image` | ❌ | Plain `<img>` everywhere |
| Routing | ❌ | Single page + hash anchors; the only second "route" is the embedded Studio |
| `next/font` | ✅ minor | Inter via Google; trivially replaced by a font link/`@font-face` |
| SEO/metadata | ⚠️ minimal | Static title + boilerplate description — achievable in plain `index.html` |
| Embedded Studio | ✅ | The one real use — but a standalone studio already exists in `sanity/` and can be hosted free at `*.sanity.studio` via `sanity deploy` |

### Recommendation: **migrate to Vite + React 19 SPA** (Option A)

This site is a static, single-page, client-animated portfolio reading from a public
CMS dataset. Next.js is adding build complexity, five serverless functions, env-var
coupling, and its own critical CVEs — while none of its differentiating features are
in use.

**Option A — Vite + React 19 + TypeScript (recommended)**
- Port the 10 components 1:1 (they're already all `'use client'` — pure client components).
- Fetch content directly with `@sanity/client` + GROQ (delete the 5 API routes and the `NEXT_PUBLIC_BASE_URL` coupling entirely). Optionally fetch at build time into static JSON for instant paint.
- Keep `sanity/` as the standalone Studio project (deployed via `sanity deploy`) — this drops `sanity`, `styled-components`, and `@sanity/vision` from the app bundle and removes most audit findings wholesale.
- SEO: full static `index.html` meta (title, description, OG/Twitter tags — an upgrade over today), plus optional prerendering if we ever want content in initial HTML.
- Deploys on Vercel as a static build (`vite build` → `dist/`): no functions, no Node version pinning, near-instant builds.
- Trade-offs: content arrives client-side (brief loading state vs. Next SSG's content-in-HTML; today's site already renders everything behind client animations, so visually nothing changes), and we own meta tags manually.

**Option B — modernize Next.js in place (Next 16 + React 19, Turbopack)**
- Keep zero-config Vercel + embedded Studio; query Sanity directly from the server component with SSG/ISR (removing the API routes anyway).
- Trade-offs: keeps the heaviest dependency chain (Studio inside the app bundle → styled-components stays, Sanity/Next peer-version treadmill continues), slower builds, more moving parts for a one-page site. React 19 + Sanity Studio embedding has historically lagged on peer support.

**Bottom line:** Option A gives a smaller bundle, faster builds, a clean `npm audit`
by construction, and simpler mental model — with zero loss of functionality or visual
identity. Option B is less migration work up front but keeps paying the complexity
tax. **Recommendation: Option A. Awaiting your call before any code is written.**
